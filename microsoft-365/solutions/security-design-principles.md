# <a name="security-hurdles-you-can-sail-overone-architects-viewpoint"></a>可以跨越的安全障碍 - 一个架构师的见解

在本文中，Microsoft 网络安全架构师 [Kozeta Garrett](https://www.linkedin.com/in/kozeta-garrett-53013a6/)描述了她在企业组织中遇到的主要安全挑战，并推荐了应对这些障碍的方法。

## <a name="about-the-author"></a>作者简介

![Kozeta Garrett 照片](../media/solutions-architecture-center/kozeta-garrett-security.jpg)

作为云安全架构师，我一直与多个组织合作，以提供战略和技术指导，专注于为迁移到 Microsoft 365 和 Azure 的客户设计和实现安全体系结构、开发企业安全解决方案，以及帮助转换安全体系结构和业务恢复文化。 我的经验包括事件检测和响应、恶意软件分析、渗透测试，以及建议改进 IT 安全性和防御状态。 我十分关心导致安全性成为企业启用器的主要转换，包括现代化工作。

了解过去两年采用安全现代化意识的组织如何处于一个很好的位置，这使它们能够继续以安全方式远程运行，尽管最近的 COVID-19 情况如何。 遗憾的是，这些情况还充当了一些客户唤醒的呼叫，这些客户尚未准备好满足此即时需求。 许多组织都意识到，他们必须快速现代化、停用累积的 IT 安全负担，并夜间改进其安全状况，以便它们可以在这些异常情况下运行。

好消息是，Microsoft 已提供一些出色的资源，可帮助组织快速提升其安全状况。 除了这些资源之外，我想分享我每天与客户遇到的主要挑战，这些挑战可以克服这些障碍。

我目前居住在北维那州，靠近我们国家/地区的华盛顿州。 我十分喜欢每一种形式的健身活动和练习，如运行、步行、步行和运动。 为了应对这些，我享受着一样多的咖啡、咖啡和旅行。

## <a name="partner-with-the-security-team-from-the-start-of-cloud-adoption"></a>从云采用开始就与安全团队合作

首先，我无法着重强调贵组织中团队从一开始进行协调的重要。 必须在云采用和设计的早期阶段将安全团队视为关键合作伙伴。 这意味着让安全团队上手支持云采用，不仅适用于向业务 (添加的功能，例如来自安全移动设备的出色的用户体验、完整的功能应用程序，或者为除有限功能电子邮件和生产力应用程序) 之外的企业数据创造价值，还利用存储、AI 和计算分析功能来帮助解决新的和旧的安全挑战。 必须包括安全团队来管理这一转变的各个方面，包括人员 (文化) 、 (培训) 和技术。 它还意味着投资现代化和持续改进安全运营中心 (SOC) 。 协同工作，使安全策略与业务策略和环境趋势保持一致，以确保数字转换安全完成。 完成此操作后，组织将开发更快地适应更改的功能，包括业务、IT 和安全性更改。

当运营和 SOC 团队之间没有真正的合作关系时，我可以看到客户遇到的问题最多。 尽管运营团队受到压力，并规定有严格的期限来采用云，但安全团队并不总是在修订和规划全面的安全策略的过程中提前参与。 这包括集成不同的云组件和部署组件。 这种缺乏合作关系会进一步流到似乎在孤立中工作以实施特定组件控制的不同团队，从而导致实现、疑难解答和集成的复杂性增加。

克服这些障碍的客户在运营和治理以及安全与风险管理团队之间建立了良好的合作关系，以修改安全策略和保护混合云工作负载的要求。 它们根据网络安全治理、风险和合规性要求，重点关注最终安全目标和结果-数据保护以及系统和服务可用性。 这些组织在运营和治理团队与 SOC 之间建立早期合作关系，这一点对于安全设计方法至关重要，并且将最大限度地提高其投资的价值。

## <a name="build-a-modern-identity-based-security-perimeter"></a>构建基于 (的新式) 安全外围

接下来，采用零信任体系结构方法。 首先构建基于身份的新式安全外围。 设计安全体系结构，其中每次访问尝试（无论是 On-prem 还是云）都将被视为不受信任的，直到验证它，"从不信任，始终验证"。 这种设计方法不仅提高了安全性和工作效率，还允许用户从任何设备类型的任何位置工作。 Microsoft 365 中包含的复杂云控件可帮助你保护用户的身份，同时根据用户风险级别控制对重要资源的访问。

有关建议的配置，请参阅 [标识和设备访问配置](../security/office-365-security/microsoft-365-policies-configurations.md)。

## <a name="transition-security-controls-to-the-cloud"></a>将安全控件转换为云

许多安全团队仍在使用针对所有本地环境构建的传统安全最佳实践，包括维护"网络外围安全"和尝试将本地安全工具和控件"强制"到云解决方案。 此类控件并非针对云设计的，无效，并且会阻碍现代云功能的采用。 适用于网络外围安全方法的过程和工具已证明效率低下，阻碍云功能，并且不允许利用新式和自动化安全功能。

你可以将防御策略转移到云托管保护、自动调查和修正、自动笔测试、Defender for Office 365 和事件分析，来克服这一障碍。 使用新式设备管理解决方案的客户在所有设备上实现了自动化管理、标准化修补、防病毒、策略实施和应用程序保护 (无论是智能手机、个人计算机、笔记本电脑还是平板电脑) 。 这样就无需使用 VPN、Microsoft System Center Configuration Manager (SCCM) 和 Active Directory 组策略。 这一点与条件访问策略相结合，可提供强大的控制和可见性，并简化了对资源的访问，无论用户从何处操作。

## <a name="strive-for-best-together-security-tools"></a>努力实现"最佳协作"安全工具

我所见，客户所克服的另一个障碍是采用"最佳方法"安全工具。 不断分层"最佳"点解决方案以满足新出现的安全需求会导致企业安全崩溃。 即使有最佳意图，大多数环境中的工具也无法集成，因为它变得过于昂贵和复杂。 这反过来又在可见性方面产生差异，因为要分类的警报多于团队可以处理的警报数。 利用新工具重新培训 SecOps 团队也成为一项挑战。

"简单更好"方法还适用于安全性。 通过采用与默认协同工作的工具的"最佳协作"策略，而不是采用"最佳协作"工具来克服这一障碍。 Microsoft 安全性功能通过跨应用程序、用户和云的集成威胁防护来保护整个组织。 通过集成，组织可以通过在进入时包含攻击者并快速修正攻击来更具弹性并降低风险。

## <a name="balance-security-with-functionality"></a>平衡安全性与功能

由于我来自一个很长的网络安全背景和体验，我倾向于从开箱即用最安全的配置开始，并允许组织根据组织的操作和安全需求来放松安全配置。 但是，这可能会以丢失功能和较差的用户体验所付出的非常代价来付出。 正如许多组织所了解的，如果安全对于用户来说太困难，他们将找到一种方法来解决你的问题，包括使用非托管云服务。 虽然我很难接受，但我已经意识到必须实现精细的功能安全平衡。

意识到用户将采取一切操作完成工作的组织确认，"影子 IT 冲突"不值得进行格斗。 他们意识到，IT 员工是影子 IT 部门以及使用未批准的 SaaS 应用程序完成工作的最大障碍。 他们转变了策略，鼓励其 (策略，而不是) 并专注于降低它可能导致的风险。 该组织的安全团队不会坚持阻止、记录所有内容，并通过反向代理或 VPN 发送所有内容。 相反，这些安全团队会减少努力来保护有价值的和敏感数据，防止向错误方或恶意应用公开。 它们努力保护数据的完整性。 他们将充分利用更高级的云信息保护功能，包括加密、安全多重身份验证、自动化风险和合规性以及云应用安全代理 (CASB) 功能，同时允许甚至鼓励跨多个平台进行受保护的共享。 他们将影子 IT 转变为激发创造力、生产力和协作，从而让企业保持竞争优势。

## <a name="adopt-a-methodical-approach"></a>采用有条理的方法

我在不同组织（无论行业如何）实施云安全方面遇到的主要挑战非常相似。 首先，尽管有大量有关特定功能和功能的丰富文档，但组织一级对适用于它们、安全功能重叠以及功能应集成如何集成存在一些混乱。 对于哪些安全功能是开箱即用预配置的，以及哪些功能需要组织进行配置，还有一些不确定。 此外，遗憾的是，SOC 团队尚未获得为准备快速采用云和数字转型所需的全面曝光、培训或预算分配，其组织已在进行。
